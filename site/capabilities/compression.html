Drift-Based Regime Detection

Traditional compression treats each value independently. VectorScaleDB understands that entity behavior exists in regimes — long periods of stability punctuated by meaningful transitions.

Cosine Drift
Measuring behavioral change in real time
As each new vector arrives, the compression engine computes the cosine distance between the incoming vector and the current segment's centroid. When this drift exceeds the configured threshold, the engine recognizes a regime change and seals the current segment.
  • Cosine distance between incoming vector and segment centroid
  • Running drift accumulator with exponential decay
  • Configurable drift threshold per entity type
  • Sub-microsecond per-vector evaluation overhead
Regime Boundaries
Semantically meaningful segmentation
Unlike fixed-window compression that splits data at arbitrary time boundaries, drift-based detection creates segments that align with actual behavioral transitions. A vehicle cruising on a highway for 3 hours produces one segment. The moment it exits and enters city traffic, a new segment begins.
  • Segments align with real behavioral changes
  • No information loss at arbitrary window boundaries
  • Variable-length segments adapt to data characteristics
  • Regime boundaries become queryable events themselves

100-1000x Compression Ratios

How VectorScaleDB achieves compression ratios that seem impossible — by exploiting the structure of real-world behavioral data.

The Insight
Most data points are redundant
In real-world temporal data, entities spend 95-99% of their time in stable behavioral regimes. A sensor reporting 1,000 vectors per second during a stable period is generating 1,000 near-identical data points. VectorScaleDB stores one segment instead.
The Math
Segment vs. raw storage
A 128-dim vector at float32 is 512 bytes. At 10 Hz over 1 hour of stable behavior, that is 36,000 vectors (18 MB). VectorScaleDB stores this as one segment: centroid (512 bytes) + start/end vectors (1,024 bytes) + metadata (256 bytes) = 1,792 bytes. Ratio: 10,000x.
Real-World
Conservative 100-1000x in production
Real entities exhibit regime changes, so actual compression varies. Highly dynamic entities (trading algorithms) see 100x. Stable entities (infrastructure sensors) see 1,000x+. The average across diverse workloads is 300-500x.

Segment Representation

Each compressed segment captures the full behavioral signature of a regime, preserving everything needed for downstream queries.

Segment Fields
What a segment contains
  • Centroid vector: Mean of all vectors in the regime
  • Start vector: First vector in the regime
  • End vector: Last vector in the regime
  • Time range: Start and end timestamps
  • Point count: Number of raw vectors compressed
  • Max drift: Maximum cosine distance observed
  • Mean drift: Average drift across the regime
  • Variance: Behavioral stability metric
Query Compatibility
Segments are first-class query targets
Compressed segments are not just storage artifacts — they are indexed in the TST and participate in all query types. KNN search matches against segment centroids. Trajectory similarity uses start/end vectors for alignment. Anomaly scoring uses drift metrics as features.
  • Segment centroids indexed in the TST for similarity search
  • Start/end vectors enable trajectory reconstruction
  • Drift metrics power anomaly detection
  • Point counts inform confidence scoring

WAL + Crash Recovery

Every write is durable before acknowledgment. The compression engine never loses data, even during unexpected failures.

Write-Ahead Log
Durability before compression
Raw vectors are appended to a write-ahead log before entering the compression pipeline. The WAL is fsync'd to disk on configurable intervals (per-write, per-batch, or timed). Only after WAL persistence does the ingest endpoint acknowledge the write.
Recovery
Replay and resume
On restart, the compression engine reads the WAL from the last checkpoint, replays un-compressed vectors through the drift detector, and resumes normal operation. In-progress segments are reconstructed from the WAL entries, ensuring no data loss and no duplicate segments.

Adaptive Threshold Tuning

Different entity types have fundamentally different behavioral characteristics. A single drift threshold does not fit all.

How It Works
Per-entity-type adaptive thresholds
VectorScaleDB maintains per-entity-type drift statistics and automatically adjusts compression thresholds to achieve a target compression ratio. High-variance entity types (e.g., financial instruments) get wider thresholds to avoid over-segmentation. Low-variance types (e.g., infrastructure sensors) get tighter thresholds to capture subtle anomalies. You set the target; the engine tunes itself.
# Python: Ingest vectors and check compression stats
import vectorscaledb

client = vectorscaledb.Client("https://your-instance.vectorscaledb.com")

# Ingest a batch of temporal vectors
response = client.ingest(
    entity_id="sensor-north-wing-03",
    entity_type="INFRASTRUCTURE",
    vectors=[
        {
            "timestamp": "2026-03-09T10:00:00.000Z",
            "embedding": [0.11, -0.22, 0.33, 0.44, -0.55, 0.66],
            "metadata": {"source": "vibration"}
        },
        {
            "timestamp": "2026-03-09T10:00:00.100Z",
            "embedding": [0.12, -0.21, 0.34, 0.43, -0.54, 0.67],
            "metadata": {"source": "vibration"}
        }
        # ... thousands more vectors
    ]
)

print(f"Ingested: {response.vectors_accepted} vectors")
print(f"Active segments: {response.active_segments}")

# Check compression statistics for this entity type
stats = client.compression.stats(entity_type="INFRASTRUCTURE")

print(f"Compression ratio:     {stats.compression_ratio}x")
print(f"Total vectors ingested: {stats.total_vectors}")
print(f"Total segments stored:  {stats.total_segments}")
print(f"Current drift threshold: {stats.drift_threshold:.4f}")
print(f"Avg segment duration:  {stats.avg_segment_duration}")

# Example output:
# Compression ratio:     847x
# Total vectors ingested: 12,400,000
# Total segments stored:  14,639
# Current drift threshold: 0.0312
# Avg segment duration:  4m 22s
